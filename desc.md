💡 What: Replaced 5 sequential, independent database queries with `Promise.all` to run concurrently, and pushed store ID filtering logic out of in-memory array manipulation directly into SQL `WHERE IN (...)` clauses for both stores and shifts. Server startup was also tweaked to allow safe test imports via `if (require.main === module)`.
🎯 Why: Sequential queries create a 'waterfall' of latency, slowing down application load times. Fetching all DB rows just to filter them in Node memory consumes excess server RAM and significantly inflates the amount of unnecessary data transferred across the network.
📊 Impact: Expected drastic reduction in overall DB latency for the initial `/api/data` app load (bound by the longest individual query rather than the sum). Significantly lower memory overhead and database I/O for large `shifts` datasets when filtering by allowed stores.
🔬 Measurement: Verify with `npm test`, testing the `/api/data` endpoint, and observing reduced query response times in browser devtools.
